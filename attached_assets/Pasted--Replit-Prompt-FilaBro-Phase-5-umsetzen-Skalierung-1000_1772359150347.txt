## Ausgangslage
- SpoolRepository.getAllView() und ggf. getByLocalIdView() haben historisch N+1 oder mehrere Roundtrips.

### Schritt 5.2: SpoolRepository View Queries ohne N+1
Update /src/data/repositories/SpoolRepository.ts:

Ersetze getAllView() und getByLocalIdView() so, dass sie:
- In exakt einer DB Query alle benötigten Daten holen, bevorzugt via LEFT JOIN:
  spools
  left join filaments
  left join manufacturers
  left join spool_stats
- Rückgabe ist SpoolView mit:
  filamentName, manufacturerName etc.
  remainingWeight aus spool_stats falls vorhanden, sonst fallback legacy spools.remaining_weight.

Wenn Drizzle Relations im Projekt sinnvoll vorhanden sind, nutze db.select + leftJoin.
Wenn nicht, baue explicit joins.

Wichtig:
- Keine Schleifen mit await pro Spule.
- Keine per Spule getByLocalId für filament/manufacturer.

### Schritt 5.4: Scan Lookup Methoden
1. Port erweitern:
   - ISpoolRepository:
     - findByQrCode(qr: string): Promise<SpoolView | null>
     - findByNfcTagId(tagId: string): Promise<SpoolView | null>

2. Implementiere in SpoolRepository über indexed WHERE.
3. Stelle sicher, dass QR und NFC Felder in DB als TEXT indexiert sind.
4. Passe Scan Flow so an, dass er diese Methoden nutzt statt Listen-Filter.
   Minimal nur die Stellen ändern, wo aktuell getAll() + filter oder getAllView() gesucht wird.

### Schritt 5.5: Sync Pull Batch Upserts
Update /src/core/application/SyncUseCase.ts und Adapter Repos:

Ziel:
- Pull remote lists (manufacturers, filaments, spools) werden in Batches upserted.
- Keine 1000 Einzel INSERT/UPDATE Statements.

Ansatz:
1. ManufacturerRepository:
   - upsertManyFromRemote(remoteList) in einer Transaktion
2. FilamentRepository:
   - upsertManyFromRemote(remoteList)
3. SpoolRepository:
   - insertManyFromRemote(remoteList) für neue local records
   - applyManyRemoteUpdates(remoteList) nur für synced records
   - Konfliktpfad bleibt per entity decision, aber Persistenz kann gesammelt werden.

Wichtig:
- Konfliktentscheidungen bleiben im UseCase.
- Persistenzoperationen dürfen batch sein.
- Bei Drizzle: insert(values).onConflictDoUpdate oder sqlite "insert or replace" je nach Schema.

  - onEndReached lädt nächste Seite
  - state appendet
