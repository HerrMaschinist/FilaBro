# Replit Prompt: FilaBro Phase 4 umsetzen (Verbrauch als Event Store + Projektion)

Du arbeitest im bestehenden FilaBro Repo. Phase 1 bis Phase 3 sind bereits umgesetzt und bauen clean.
Phase 3 Ergebnis:
- Offline First Konflikte sind implementiert, keine Server Wins Logik.
- conflict_snapshots Tabelle existiert, ConflictSnapshotRepository existiert.
- Domain ist rein unter /src/core/domain.

Jetzt implementierst Du Phase 4: Verbrauchsmodell als append-only Events, mit skalierbarer Projektion.

## Ziele Phase 4
1. Verbrauch wird als Ereignisse gespeichert, append-only.
2. remainingWeight wird nicht mehr als “Wahrheit” mutiert, sondern aus Events berechnet oder aus Projektion gelesen.
3. Audit Trail ist vorhanden.
4. Performance bleibt gut bei 1000+ Spulen.
5. Nach Phase 4 baut das Projekt inkl. tsc.

## Nicht verhandelbar
1. Konkrete Dateipfade, kompletter Dateiinhalt oder Diff pro Änderung.
2. Keine UI Redesigns. Nur minimale Anpassungen, damit bestehende Funktionen weiterlaufen.
3. Domain bleibt rein. Keine Infrastruktur Imports in /src/core/domain.
4. Keine Datenverlust Migration. Bestehende Werte bleiben nutzbar.
5. Jede neue Tabelle bekommt sinnvolle Indizes.

## Ausgangslage
- Spool hat Felder wie remainingWeight, initialWeight, spoolWeight, usedWeight (oder ähnlich).
- Aktuell wird remainingWeight direkt aktualisiert (in Repo oder UseCase).
- Verbrauchsassistenz braucht Verlauf, Undo, Prognosen. Dafür Event Store.

## Phase 4 Aufgaben

### Schritt 4.1: Domain Typen für Usage Events anlegen
Erstelle:
- /src/core/domain/usage/types.ts
- /src/core/domain/usage/index.ts
- Update /src/core/domain/index.ts, damit usage exportiert wird.

Inhalt:
- UsageEventType = "consume" | "adjustment"
- UsageEventSource = "manual" | "scan" | "sync" | "import"
- UsageEvent:
  - id: string
  - spoolLocalId: string
  - grams: number
  - type: UsageEventType
  - occurredAt: number (unix ms)
  - source: UsageEventSource
  - note?: string

Regeln:
- grams ist positiv.
- consume reduziert Rest, adjustment kann hoch oder runter, aber wird als grams mit type adjustment gespeichert.
- Domain Dateien dürfen keine DB Imports haben.

### Schritt 4.6: Application Use Case für Verbrauch implementieren
Update oder neu:
- /src/core/application/SpoolUseCase.ts oder /src/core/application/WeightUseCase.ts

Neue Methode:
- recordUsage(spoolLocalId: string, grams: number, source: UsageEventSource, note?: string)

Ablauf:
1. Hole aktuelle Remaining Projection aus spool_stats.
2. Ermittle Ausgangswert:
   - Wenn spool_stats.remaining_weight definiert, nutze den
   - Sonst fallback:
     - Wenn Spool.initialWeight und Spool.spoolWeight existieren, dann initialRemaining = initialWeight
     - Wenn Spool.remainingWeight historisch existiert, nutze das als Startwert
3. Append UsageEvent (type consume).
4. Berechne neuen remaining = max(0, oldRemaining - grams)
5. Upsert spool_stats mit remaining und updatedAt.

Wichtig:
- Für Korrekturen: adjustment Events erlauben. Dazu zweite Methode:
  - adjustRemaining(spoolLocalId: string, newRemaining: number, source, note?)
  - Implementiere als adjustment Event, und setze spool_stats direkt auf newRemaining.

### Schritt 4.8: Sync Verhalten für Verbrauch definieren
Regel für Phase 4:
- Sync verändert keine usage_events.
- Spoolman Push/Pull bleibt wie Phase 3, aber remainingWeight kommt jetzt aus spool_stats.
- Wenn remote remainingWeight kommt:
  - Bei accept_remote und lokal synced: setze spool_stats auf remote remainingWeight, aber schreibe zusätzlich einen adjustment Event mit source sync.
  - Bei conflict: kein Überschreiben.

Das ist wichtig für Audit Trail.
